## 문제와 해결 과정

### 1차 시도: 기본 설정 — 실패
```
vllm serve Qwen/Qwen2.5-1.5B-Instruct --device cpu --dtype auto
```
- `--device cpu` → v0.17.1에서 **인자 제거됨** (unrecognized arguments)
- `VLLM_USE_V1=0` → v0.17.1에서 **환경변수 제거됨** (Unknown variable)

### 2차 시도: 인자 수정 — EngineCore 크래시
```
vllm serve Qwen/Qwen2.5-0.5B-Instruct --dtype bfloat16 --max-model-len 256
```
- V1 EngineCore 서브프로세스가 spawn 후 즉시 사망
- `RuntimeError: Engine core initialization failed. Failed core proc(s): {}`
- 원인: EngineCore가 별도 프로세스로 뜨면서 동일한 SIGILL 발생

### 3차 시도: enforce_eager 추가 — 여전히 SIGILL
```
vllm serve ... --dtype bfloat16 --enforce-eager
VLLM_ENABLE_V1_MULTIPROCESSING=0
```
- torch.compile 비활성화 → CompilationMode.NONE 확인
- 모델 가중치 로딩 성공 (0.13초)
- **KV cache 초기화 단계에서 SIGILL (exit code 132)**

### 원인 특정: BF16 명령어 비호환
```python
# PyTorch 기본 테스트
t = torch.randn(10, 10)
t @ t                          # FP32 → 성공
t.to(torch.bfloat16) @ t_bf16  # BF16 → 성공 (단순 matmul)

# transformers 직접 추론
model(..., dtype=torch.bfloat16)  # SIGILL ← 대규모 BF16 연산에서 발생
model(..., dtype=torch.float32)   # 성공
```
- VirtualBox ARM64 VM이 일부 ARM BF16 확장 명령어를 패스스루하지 못함
- 단순 BF16 연산은 되지만, 대규모 attention 연산에서 특정 SIMD 경로 진입 시 SIGILL

### 최종 해결: FP32 + eager + 단일프로세스
```
vllm serve Qwen/Qwen2.5-0.5B-Instruct \
  --dtype float32 \
  --max-model-len 256 \
  --enforce-eager

환경변수:
  VLLM_ENABLE_V1_MULTIPROCESSING=0
  VLLM_CPU_KVCACHE_SPACE=1
  VLLM_TARGET_DEVICE=cpu
```

## 성공 조건 요약

| 설정 | 값 | 이유 |
|------|-----|------|
| `--dtype` | `float32` | BF16이 VirtualBox ARM64 VM에서 SIGILL 유발 |
| `--enforce-eager` | 필수 | torch.compile이 비호환 명령어 사용 |
| `VLLM_ENABLE_V1_MULTIPROCESSING` | `0` | EngineCore 서브프로세스 spawn 실패 방지 |
| `--max-model-len` | `256` | 메모리 절약 (KV cache 크기 제한) |
| `VLLM_CPU_KVCACHE_SPACE` | `1` (GB) | KV Cache 총 크기 제한 |

| | 0.5B | 1.5B |

| | 0.5B | 1.5B |
|--|------|------|

```

```

| 호스트 메모리 | 워커 메모리 | VM 합계 | 실습 범위 |
|-------------|-----------|---------|----------|
| **32GB** | 8GB x 3 | 28GB | 7.1~7.4 (vLLM 제외) |
| **48GB** | 12GB x 3 | 40GB | 7.5 가능 (Qwen2.5-0.5B/1.5B) |
| **64GB** | 16GB x 3 | 52GB | 7.5 최신 모델 (Qwen3-1.7B) |

### 7.5.3 — MoA로 sLLM의 한계 극복하기
Mixture of Agents 패턴: 3개 모델에 동일 질문 → aggregator가 종합 → 더 나은 답변.

실습 파일:
```
ch7/7.5.3/
├── moa-aggregator.yaml          # MoA 스크립트(ConfigMap) + 영어 테스트 Pod
├── moa-korean.yaml              # 한국어 MoA 테스트 Pod
└── moa-change-aggregator.yaml   # Aggregator 변경 비교 (Qwen vs Gemma)
```

실습 순서:
```bash
# 1. MoA 스크립트 배포 + 영어 테스트
kubectl apply -f moa-aggregator.yaml
kubectl logs moa-english       # 개별 응답 + MoA 종합 확인

# 2. 한국어 MoA 테스트
kubectl apply -f moa-korean.yaml
kubectl logs moa-korean

# 3. Aggregator 변경 비교
kubectl apply -f moa-change-aggregator.yaml
kubectl logs moa-qwen-agg     # Qwen3.5가 aggregator
kubectl logs moa-gemma-agg    # Gemma3가 aggregator
# → Aggregator 선택에 따라 품질이 달라짐을 확인

# 4. (선택) Hubble로 Pod 간 트래픽 흐름 관찰
#    → 7.2.2에서 배포한 Hubble 활용
hubble observe --namespace default
```

관찰 포인트:
1. 개별 답변 vs MoA 종합 답변 → **종합이 더 나음** (한국어 Docker vs K8s)
2. Aggregator가 할루시네이션 모델이면 → **오류 전파** (영어 Qwen3.5 aggregator)
3. Hubble에서 moa-pod → 3개 ollama 서비스로의 트래픽 확인

### ARM64 CPU에서 양자화 지원 현황

| 양자화 방식 | x86 CPU | ARM64 CPU | 비고 |
|------------|---------|-----------|------|
| **GPTQ** | O | **X** | x86 전용 커널 |
| **AWQ** | O | **X** | x86 전용 커널 |
| **GGUF** | △ | **X** | GPU용 설계, CPU 커널 없음 (llama.cpp 미사용) |

### 결론: vLLM에서 Q4는 ARM64 VM에서 불가

- GPTQ/AWQ: x86 전용, ARM64에서 로드 자체가 안 됨

- compressed-tensors: Graviton4 + KleidiAI/ACL 필요, VirtualBox VM 불가
- **Q4 양자화 추론은 Ollama(llama.cpp 기반)에서만 가능**

## 참고사항

### 서비스 이름 주의
- K8s Service 이름에 `vllm-*`을 사용하면 K8s가 `VLLM_*` 환경변수를 자동 주입
- vLLM이 이를 자체 환경변수로 오인하여 경고 발생
- 서비스 이름을 `llm-serve-*` 등으로 변경하면 해결

### vLLM 참고용 코드 (48GB+ 호스트)
- vLLM은 ARM64 VM에서 FP32 강제 (Q4 불가)
- 성공 조건: `--dtype float32`, `--enforce-eager`, `VLLM_ENABLE_V1_MULTIPROCESSING=0`
- 48GB+ 호스트용 참고 코드만 제공, 책 기본 실습은 Ollama

### 최신 모델 사용
- 책 기본: **Qwen3.5(2026.03), Gemma3(2025.03), Llama3.2(2024.09)** — 글로벌 3대 AI 기업
- `sysnet4admin/ollama-*` 커스텀 이미지 빌드 필요
- Gemma3 이미지 빌드 시 HF 토큰 필요 (Ollama 실행 시에는 불필요)

## MoA (Mixture of Agents) 용어 정리

### 출처
- **논문**: "Mixture-of-Agents Enhances Large Language Model Capabilities"
- **저자**: Junlin Wang, Jue Wang, Ben Athiwaratkun, Ce Zhang, James Zou (Together AI)
- **발표**: 2024.06 (arXiv:2406.04692), **ICLR 2025 accept**
- **GitHub**: github.com/togethercomputer/MoA

### 핵심 개념
여러 LLM을 레이어 구조로 배치 → 각 레이어의 에이전트가 이전 레이어 출력을 참고하여 정제 → 최종 답변.
오픈소스 MoA가 AlpacaEval 2.0에서 GPT-4o를 초과 (65.1% vs 57.5%).

### 동음이의어 주의 (책에서 풀네임 병기 필요)

| 약어 | 의미 | 분야 |
|------|------|------|
| **MoA** | **Mixture of Agents** | 다중 LLM 협업 (**7.5.3에서 사용**) |
| MoA | Mixture of Attention Heads | 어텐션 아키텍처 |
| MoA | Mixture of Sparse Attention | LLM 압축 |

### 유사 패턴과의 차이

| 패턴 | MoA와의 차이 |
|------|-------------|
| Multi-Agent Debate | 에이전트가 서로 비판 (adversarial), MoA는 레이어 정제 |
| LLM Ensemble | 다수결 투표, MoA는 상호 참조 후 종합 |
| Self-Consistency | 단일 모델 다중 샘플링, MoA는 다중 모델 |

### 7.5.3과의 관계
7.5.3에서 구현하는 것은 MoA의 **단순화 버전** (1레이어: 3개 모델 응답 → 1개 aggregator 종합).
논문 원본은 다중 레이어이지만, sLLM 협업의 개념 전달에는 충분함.

### 핵심 결론

1. **opt-w12g-ollama가 가장 우수**: Q4 양자화로 큰 모델을 적은 메모리로 구동, 영어/한국어 모두 높은 품질

5. **Thinking 모델 주의**: 소형(0.8B~2B)에서는 thinking에 대부분 토큰 소비, 4B+에서 효과적
6. **SmolLM2-1.7B (vLLM FP32)**: 첫 로딩 시 OOMKilled 발생 → 12GB 워커에서 불안정
