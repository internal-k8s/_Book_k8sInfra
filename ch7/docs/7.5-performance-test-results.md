## 시나리오 3: opt-w12g-vllm (48GB 랩탑, Worker 12GB, vLLM/OpenAI API)

| 모델 | 노드 | 런타임 | 메모리 사용 | 비고 |
|------|------|--------|------------|------|
| Qwen2.5-0.5B-Instruct | w1-k8s | vLLM (FP32) | 6.0GB / 11GB | |
| Qwen2.5-1.5B-Instruct | w2-k8s | vLLM (FP32) | 10GB / 11GB | |
| SmolLM2-1.7B-Instruct | w3-k8s | vLLM (FP32) | 10GB / 11GB | OOMKilled 1회 발생 |

⚠️ **vLLM 제약사항**:
- FP32 전용 (ARM64 VirtualBox에서 BF16 불가 — SIGILL 발생)
- `--max-model-len 256` (메모리 제약으로 컨텍스트 길이 제한)
- 한국어 토큰이 영어보다 길어 max-model-len 초과 가능
- 2024년 모델 사용 (2026년 모델은 vLLM 호환 미확인)

- 메모리 여유가 거의 없는 상태로 운영

### MoA 테스트 (opt-w12g-vllm)

#### 영어 MoA (Qwen2.5-1.5B 집계)

- max-model-len 256 제한으로 전체 응답을 프롬프트에 포함 불가 → 요약된 짧은 프롬프트 사용

#### 한국어 MoA (Qwen2.5-1.5B 집계)
> "Kubernetes (K8s)은 컨테이너 관리 플랫폼이며, 배포 자동화와 오케스트레이션 시스템이다."
- 품질: ★★☆ (간결하지만 깊이 부족)
- 컨텍스트 길이 제한이 MoA 품질에 직접적 영향

### opt-w12g-vllm 시나리오 결론
- vLLM은 OpenAI 호환 API를 제공하는 장점이 있음
- 그러나 ARM64 CPU에서 FP32 필수 → **메모리 소비가 Ollama Q4 대비 ~4배**
- max-model-len 256 제한으로 MoA 같은 긴 프롬프트에 부적합
- SmolLM2-1.7B는 OOMKilled 위험 (12GB 워커에서 한계)
- 한국어 능력: Qwen2.5 계열만 부분적으로 가능, SmolLM2는 불가

### 핵심 결론

1. **opt-w12g-ollama가 가장 우수**: Q4 양자화로 큰 모델을 적은 메모리로 구동, 영어/한국어 모두 우수

5. **Thinking 모델 주의**: 소형(0.8B~2B)에서는 thinking에 대부분 토큰 소비, 큰 모델(4B+)에서 효과적
